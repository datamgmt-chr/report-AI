3.1
Tool registry pattern
Dict dispatcher ¬∑ type annotations
Non usare un if/elif gigante per decidere quale tool eseguire. Usa un registry dizionario ‚Äî ogni tool √® una funzione registrata con un decoratore. Aggiungere nuovi tool diventa triviale.

  # Decoratore per registrare tool
  def decorator(fn):
    TOOL_REGISTRY[name] = fn
    return fn
  return decorator

@tool("search_kb")

  return vector_db.search(query)

def execute_tools(content_blocks: list):
  return [
    TOOL_REGISTRY[b.name](**b.input)
    for b in content_blocks
    if b.type == "tool_use"
  ]

04Trappole tipiche e come evitarle
‚öë Errori frequenti nello stack Custom
Trappola
Come evitarla
‚úï
Loop infinito senza hard limit ‚Äî il modello continua a chiamare tool senza convergere verso una risposta finale.
Imposta sempre max_iterations come parametro esplicito. Logga ogni iterazione. Se raggiungi il limite, solleva un'eccezione specifica, non silenziosa.
‚úï
Schema tool non abbastanza precisi ‚Äî tool descritti vagamente vengono usati nel momento sbagliato o con argomenti errati.
Usa Pydantic per definire schema rigorosi. Aggiungi esempi nella description del tool. Testa ogni tool in isolamento prima di integrarlo nel loop.
‚úï
Stato della conversazione in memoria ‚Äî la lista messages cresce a ogni iterazione e non viene mai troncata.
Implementa una strategia di context management: mantieni solo gli ultimi N messaggi, o usa summarization periodica per comprimere la storia.
‚úï
Eccezioni dei tool inghiottite silenziosamente ‚Äî un tool fallisce, l'agente riceve un risultato vuoto e continua come se nulla fosse.
Ogni tool deve restituire un errore strutturato al modello quando fallisce. Il modello deve "vedere" l'errore per poter reagire correttamente.

03Trappole tipiche ‚Äî Multi-agent
‚öë Errori frequenti nei sistemi Multi-agent
Trappola
Come evitarla
‚úï
Aggiungere agenti senza misurare il beneficio ‚Äî ogni agente aggiunto moltiplica latenza e costi senza necessariamente migliorare la qualit√†.
Misura la qualit√† dell'output single-agent sul tuo eval set. Aggiungi un agente solo se il delta di qualit√† giustifica il delta di costo e latenza. Documenta la decisione.
‚úï
Errori che si propagano silenziosamente ‚Äî agente A produce output errato, agente B lo usa come input, agente C produce output plausibile ma sbagliato.
Valida gli output tra un agente e l'altro. Usa output strutturati (Pydantic) con schema rigidi che falliscono esplicitamente se l'agente precedente ha prodotto dati malformati.
‚úï
Costi incontrollati ‚Äî 3 agenti con 5 tool calls ciascuno a 8 iterazioni = 120 chiamate LLM per task. A $0.01/chiamata: $1.20 per task. A 1000 task/giorno: $1200/giorno.
Imposta token budget espliciti per ogni agente (max_iter, max_tokens). Monitora il costo per run fin dall'inizio. Usa modelli pi√π leggeri per agenti di reasoning semplice.
‚úï
Loop tra agenti ‚Äî agente A chiede a B di fare X, B chiede ad A di fare Y, loop infinito.
In sistemi con comunicazione bidirezionale tra agenti, implementa sempre un hard limit di messaggi totali nel crew, non solo per singolo agente.
Quando scegliere

Research + analisi + redazione. Pipeline di content con fact-checking. Sistemi dove serve un "critic" che validi l'output di altri agenti. Workflow paralleli che devono convergere.
Principale vantaggio

Ogni agente pu√≤ avere prompt, tool e anche modello diversi ottimizzati per il suo ruolo. Un agente critico che rivede l'output degli altri produce qualit√† sistematicamente migliore.
