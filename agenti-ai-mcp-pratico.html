Standard aperto · Anthropic 2024 JSON-RPC 2.0 over stdio / HTTP Client ↔ Server protocol
01 — Definizione operativa
Cosa è MCP, concretamente

MCP è uno standard che definisce come un agente AI chiede e usa strumenti esterni. Non è una libreria, non è un servizio cloud, non è un'azienda. È una specifica — come HTTP o USB — che dice: "se vuoi che un agente possa usare il tuo strumento, implementa questa interfaccia". Il risultato è che agente e strumento si parlano senza che nessuno dei due debba sapere come è fatto l'altro.

Prima di MCP
N×M integrazioni custom
Hai 3 agenti (LangChain, Claude, Cursor) e 5 strumenti (GitHub, Slack, Notion, DB, Drive). Ogni coppia agente-strumento richiede codice custom. Risultato: 15 integrazioni diverse da scrivere, mantenere, aggiornare. Quando GitHub aggiorna la sua API, devi aggiornare 3 integrazioni separatamente.
→
Con MCP
N+M implementazioni standard
GitHub implementa un server MCP una volta sola. LangChain implementa il client MCP una volta sola. Ogni agente che supporta MCP può usare automaticamente ogni strumento che supporta MCP. 3+5=8 implementazioni invece di 15. Quando GitHub aggiorna, aggiorna solo il suo server MCP.
Concetto 1
MCP Server
Il lato dello strumento. Un server MCP espone le funzionalità di un sistema (es. GitHub) in formato standard: lista degli strumenti disponibili, schema degli input/output, logica di esecuzione. Chiunque lo implementa una volta, per tutti i client MCP.
Concetto 2
MCP Client
Il lato dell'agente. Un client MCP sa come connettersi a qualsiasi server MCP, scoprire i suoi strumenti e usarli. LangChain, Claude API, Cursor — tutti implementano MCP come client e guadagnano accesso a tutti i server esistenti automaticamente.
Concetto 3
Tool Discovery
L'agente non deve sapere a priori cosa può fare un server. Chiede al server la lista degli strumenti disponibili a runtime. Il server risponde con nome, descrizione e schema di ogni tool. L'LLM decide autonomamente se e quale usare in base al task.
Cosa succede tecnicamente quando un agente usa un server MCP Flusso messaggi JSON-RPC
1
Il client si connette al server
L'agente apre una connessione al server MCP (via stdio o HTTP/SSE). Avviene una volta all'avvio.
// Connessione iniziale
client.connect("npx @modelcontextprotocol/server-github")
2
Il client chiede "che strumenti hai?"
Il client invia una richiesta tools/list. Il server risponde con la lista completa degli strumenti e i loro schemi.
// Risposta del server: lista strumenti
[
  { name: "create_issue", description: "Crea un issue su GitHub",
    inputSchema: { repo: "string", title: "string", ... } },
  { name: "list_prs", description: "Lista le PR aperte", ... }
]
3
La lista degli strumenti va nel contesto del LLM
Il framework inserisce automaticamente la descrizione degli strumenti MCP nel prompt del modello. L'LLM "vede" gli strumenti disponibili e decide se usarli.
// Cosa vede il LLM nel suo contesto
"Hai accesso ai seguenti strumenti:
- create_issue: Crea un issue su GitHub. Parametri: repo, title, body...
- list_prs: Lista le PR aperte. Parametri: repo, state..."
4
Il LLM decide di chiamare uno strumento
In risposta al task dell'utente, il modello produce una chiamata strumento strutturata invece di testo libero.
// Output del LLM (non testo, ma tool call)
{ tool: "create_issue",
  input: { repo: "mia-azienda/backend", title: "Bug nel login",
           body: "Il form non valida l'email" } }
5
Il client esegue lo strumento sul server
Il framework intercetta la tool call, la manda al server MCP con tools/call. Il server esegue l'azione reale (chiama l'API GitHub) e restituisce il risultato.
// Risposta del server dopo l'esecuzione
{ content: [{ type: "text",
  text: "Issue #42 creato: https://github.com/mia-azienda/backend/issues/42" }] }
6
Il risultato torna nel contesto del LLM
Il framework aggiunge il risultato al contesto. Il LLM continua il ragionamento con questa nuova informazione e produce la risposta finale all'utente.
// Il LLM risponde all'utente con l'esito
"Ho creato l'issue #42 'Bug nel login' su mia-azienda/backend.
Puoi trovarlo qui: [link]. Vuoi che assegni anche un assignee?"
02 — Impatto pratico
Cosa cambia concretamente
quando costruisci un agente

MCP non cambia la logica dell'agente — cambia come connetti gli strumenti. Invece di scrivere codice di integrazione, configuri una connessione a un server MCP. Questo ha impatti diretti su tempo, manutenzione e architettura.

Confronto: strumento custom vs strumento MCP Esempio pratico: connettere GitHub a un agente LangChain
❌ Senza MCP — integrazione custom (100+ righe)
# Scrivi tu tutto da zero
import requests

class GitHubTool:
  def __init__(self, token):
    self.headers = {"Authorization": f"token {token}"}
    self.base_url = "https://api.github.com"

    # gestione errori, retry, rate limiting,

    response = requests.post(url,
      headers=self.headers,
      json={"title": title, "body": body})
    if response.status_code != 201:
      raise Exception(f"Errore: {response.text}")
    return response.json()

  def list_prs(self, repo, state):
    # ... altra logica custom ...

# Wrap come tool LangChain
from langchain.tools import StructuredTool
github = GitHubTool(token=os.getenv("GITHUB_TOKEN"))
tools = [StructuredTool.from_function(github.create_issue), ...]
✓ Con MCP — 5 righe di configurazione
# Il server MCP GitHub è già scritto e mantenuto
# Tu configuri solo la connessione

from langchain_mcp_adapters.client import MultiServerMCPClient

client = MultiServerMCPClient({
  "github": {
    "command": "npx",
    "args": ["@modelcontextprotocol/server-github"],
    "env": {"GITHUB_PERSONAL_ACCESS_TOKEN": "..."}
  }
})

# Gli strumenti GitHub sono ora disponibili
# per il tuo agente. Zero altro codice.
tools = await client.get_tools()
Impatto 1 — Tempo di sviluppo
Da giorni a ore per ogni integrazione
Connettere GitHub a un agente senza MCP richiede 1-3 giorni (scrivere il tool, gestire errori, testare, documentare). Con MCP: configurare la connessione richiede 30 minuti. Il risparmio si moltiplica per ogni strumento che aggiungi. Un agente con 10 strumenti: 30 giorni → 5 ore.
Impatto 2 — Manutenzione
Aggiornamenti automatici dagli owner
Quando GitHub aggiorna la sua API, aggiorna il suo server MCP. Tu aggiorni una dipendenza npm/pip e hai subito la versione nuova — senza toccare il tuo codice. Senza MCP, dovresti aggiornare la tua integrazione custom ogni volta che cambia l'API esterna.
Impatto 3 — Portabilità
Stesso server MCP funziona su tutti gli agenti
Il server MCP che usi con LangChain funziona identicamente con Claude API, con Cursor, con Flowise. Se cambi framework agente, non devi riscrivere nessuna integrazione strumento. L'investimento nella configurazione dei server MCP è portabile tra stack diversi.
Impatto 4 — Complessità architetturale
Un processo separato per ogni server
Ogni server MCP gira come processo separato (tipicamente Node.js). Il tuo agente Python coordina N processi esterni via stdio. Questo aggiunge overhead di gestione dei processi e può complicare il deployment. È il trade-off principale da considerare.

⚠
Come valutare la qualità di un server MCP prima di usarlo
Controlla: stelle GitHub (>500 è un buon segnale), data dell'ultimo commit (non usare repo abbandonati da >6 mesi), presenza di test e documentazione, chi lo mantiene (vendor ufficiale > community attiva > singolo sviluppatore). Prima di deployare in produzione, testa ogni server MCP in ambiente isolato — alcuni server community hanno bug che causano comportamenti imprevedibili nell'agente.
04 — Supporto per stack
Come MCP si integra
nel tuo stack attuale

Il supporto MCP varia molto a seconda dello stack che stai usando. Questa è la situazione attuale — in rapida evoluzione.

Stack Supporto MCP Come si usa Note pratiche
Claude API
ANTHROPIC
 ✓ Nativo Passa server MCP nel parametro mcp_servers della chiamata API. I tool vengono esposti automaticamente al modello. Il modo più semplice. Nessuna libreria aggiuntiva. Funziona con URL di server MCP remoti (HTTP/SSE).
LangChain
FRAMEWORK
 ✓ Supportato Libreria langchain-mcp-adapters. Converte tool MCP in LangChain tools. Compatible con tutti gli agenti LangChain. Richiede pip install langchain-mcp-adapters. Server stdio girano come subprocess Python.
LangGraph
FRAMEWORK
 ✓ Supportato Stesso adapter di LangChain. I tool MCP sono nodi del grafo come qualsiasi altro tool. Ottima integrazione — i tool MCP possono essere usati in qualsiasi nodo del grafo agente.
LlamaIndex
FRAMEWORK
 ✓ Supportato Pacchetto llama-index-tools-mcp. Integrazione con agenti LlamaIndex tramite MCPToolSpec. Particolarmente utile per connettere fonti dati via MCP al pipeline RAG.
CrewAI
MULTI-AGENT
 ✓ Supportato Tool MCP diventano tool CrewAI assegnabili agli agenti. Configurazione nel codice del crew. Permette di dare a ogni agente del crew set di tool MCP diversi e specializzati.
Flowise
NO-CODE
 ~ Parziale Nodo "MCP Tool" disponibile nell'editor visivo. Configurazione del server via UI senza codice. Supporto in crescita. Non tutti i server MCP sono compatibili. Meglio testare prima.
n8n
NO-CODE
 ~ In arrivo Supporto MCP annunciato. Integrazione tramite nodi HTTP per server MCP remoti (workaround). Non ancora nativo. Per ora: usa HTTP request node per chiamare server MCP con trasporto HTTP/SSE.
Agentforce / M365 Copilot
PROPRIETARIO
 ✗ Non supportato Le piattaforme enterprise chiuse non espongono MCP. Usano i propri sistemi di connettori proprietari. Non applicabile. Con questi stack non hai controllo sull'implementazione dei tool.
Custom / Python puro
VANILLA
 ✓ Via libreria Libreria ufficiale mcp Python SDK. Implementa client e server MCP da zero nel tuo codice. Massimo controllo. Puoi costruire sia client che server MCP custom con la SDK ufficiale.
05 — Costruire server custom
Quando e come costruire
un server MCP aziendale

Se hai un sistema interno che non ha un server MCP pubblico, puoi costruirne uno. Il beneficio è trasformare il tuo sistema in uno strumento standard riutilizzabile da qualsiasi agente — anche futuri. È meno complesso di quanto sembri.

Costruisci un server MCP quando
Hai un sistema interno proprietario
CRM custom, gestionale interno, database legacy, API proprietaria. Nessun server MCP pubblico lo copre. Costruire un server MCP lo rende accessibile a tutti gli agenti aziendali, ora e in futuro.
Costruisci un server MCP quando
Vuoi esporre un agente ad altri agenti
Un agente RAG sui documenti aziendali può diventare un server MCP. Altri agenti lo chiamano come strumento — "cerca nella knowledge base". Stai costruendo un ecosistema agente modulare.
Non costruire un server MCP quando
Esiste già un server pubblico
Controlla sempre prima se esiste un server MCP ufficiale o community per lo strumento che vuoi connettere. Costruire un doppione è tempo sprecato e ti crei manutenzione inutile.
Struttura di un server MCP Python — Esempio minimo funzionante Esporre un CRM interno come strumento MCP
## server_crm.py — Server MCP per CRM aziendale interno
from mcp.server import Server, NotificationOptions
from mcp.server.models import InitializationOptions
import mcp.types as types
import mcp.server.stdio

# 1. Crea il server
server = Server("crm-aziendale")

# 2. Dichiara gli strumenti disponibili
@server.list_tools()
async def handle_list_tools() -> list[types.Tool]:
  return [
    types.Tool(
      name="cerca_cliente",
      description="Cerca un cliente nel CRM per nome o email",
      inputSchema={
        "type": "object",
        "properties": {
          "query": {"type": "string", "description": "Nome o email del cliente"}
        },
        "required": ["query"]
      }
    ),
    types.Tool(
      name="aggiorna_stato",
      description="Aggiorna lo stato di un'opportunità",
      inputSchema={ ... }
    )
  ]

# 3. Implementa l'esecuzione degli strumenti
@server.call_tool()
async def handle_call_tool(name: str, arguments: dict):
  if name == "cerca_cliente":
    # Chiama il tuo CRM interno reale
    risultati = await crm_api.search(arguments["query"])
    return [types.TextContent(type="text", text=str(risultati))]

# 4. Avvia il server (trasporto stdio)
async def main():
  async with mcp.server.stdio.stdio_server() as (read, write):
    await server.run(read, write, InitializationOptions(...))
→ Salva come server_crm.py · Configura nel tuo agente: "command": "python", "args": ["server_crm.py"]
06 — Decisioni concrete
Le 3 decisioni da prendere
su MCP per il tuo agente

Non devi "decidere se usare MCP" in astratto. Ci sono tre decisioni operative specifiche, ognuna con criteri chiari.

Decisione 1 — Usare server MCP esistenti o integrazioni custom?
Lo strumento che vuoi connettere ha un server MCP pubblico stabile?
Cerca su github.com/modelcontextprotocol/servers e su mcp.so. Verifica qualità (stelle, commit recenti, mantenuto da vendor ufficiale).
SÌ → usa il server MCP NO → valuta
✓ Usa il server MCP esistente
Configura la connessione in 30 minuti. Non scrivere codice di integrazione. Se il server è ufficiale del vendor, fidati della sua qualità. Se è community, testa prima in staging.
→ Esiste un server community con buona qualità?
Se sì: considera di usarlo ma monitora attentamente. Se no: scrivi un'integrazione custom tradizionale oppure costruisci il tuo server MCP (vedi Decisione 3).
Decisione 2 — Il tuo stack supporta MCP? Cambia qualcosa?
Stai usando o valutando LangChain, LlamaIndex, CrewAI, o la Claude API diretta?
Questi stack supportano MCP nativamente o con un adapter leggero. Puoi iniziare a usare server MCP subito.
SÌ → puoi usare MCP ora NO (es. Flowise, n8n, piattaforme enterprise)
✓ Nessuna azione necessaria sullo stack
Installa l'adapter MCP per il tuo framework. Configura i server MCP che ti servono. Lo stack è già pronto.
→ Verifica roadmap della piattaforma
Se usi piattaforme no-code, controlla se supportano MCP (Flowise in parte) o se è in roadmap. Se usi piattaforme enterprise chiuse, MCP non è applicabile — usa i connettori nativi del vendor.
Decisione 3 — Esporre i tuoi agenti e sistemi come server MCP?
Hai sistemi interni o agenti che potrebbero essere utili ad altri agenti nel tuo ecosistema?
Es: agente RAG sui documenti aziendali, CRM interno, gestionale proprietario, database di prodotto.
SÌ, più agenti devono accedere NO, solo un agente lo usa
→ Considera di costruire un server MCP
Esponi il sistema come server MCP. Ogni agente che vuole usarlo si connette allo stesso server standardizzato. Investimento iniziale: 1-3 giorni. Valore: non devi più riscrivere l'integrazione per ogni nuovo agente.
✓ Integrazione custom è sufficiente
Se solo un agente usa quel sistema, non vale la pena costruire un server MCP. Un'integrazione custom diretta è più semplice e richiede meno overhead.
07 — Checklist operativa
Prima di deployare un agente
con server MCP

MCP aggiunge complessità operativa. Questi sono i punti da verificare prima di mandare in produzione un agente che usa server MCP — molti vengono ignorati nei prototipi e diventano problemi in produzione.

Checklist di produzione — Agente con MCP
Sicurezza
I server MCP girano in sandbox con permessi minimi
Il server filesystem MCP deve accedere solo alle cartelle necessarie. Il server DB deve avere accesso read-only salvo necessità specifiche.
MUST
Le credenziali API dei server MCP sono in variabili d'ambiente, non nel codice
GITHUB_TOKEN, SLACK_BOT_TOKEN ecc. non devono mai essere hardcoded nel codice o nei file di configurazione versionati.
MUST
Hai validato che i server MCP community non eseguono codice arbitrario
Leggi il codice sorgente dei server community prima di usarli in produzione. I server ufficiali Anthropic sono sicuri; i community richiedono review.
MUST
Deployment e infrastruttura
Node.js è installato sull'ambiente di produzione
La maggior parte dei server MCP ufficiali richiede Node.js (npx). Se il tuo agente gira in container Python, aggiungi Node.js al Dockerfile.
MUST
Il processo del server MCP si riavvia automaticamente in caso di crash
Usa un process manager (systemd, supervisord, PM2) o gestisci il riavvio nel codice client. I server MCP stdio sono processi separati che possono crashare.
MUST
Hai testato cosa succede quando un server MCP è offline
L'agente deve degradare gracefully — non crashare — se un server MCP non risponde. Implementa fallback e timeout espliciti.
SHOULD
Costi e performance
Hai misurato l'impatto sul context window dell'agente
La lista degli strumenti MCP va nel prompt. 10 server con 5 tool ciascuno aggiungono ~2000 token a ogni chiamata. Con molti server, i costi aumentano significativamente.
SHOULD
Hai considerato di limitare gli strumenti esposti per ogni chiamata
Non connettere 10 server MCP a ogni singola richiesta. Carica solo i server rilevanti per il task corrente — riduce i token e riduce la probabilità che il modello usi lo strumento sbagliato.
NICE
Osservabilità
Stai loggando quali tool MCP vengono chiamati e con quali argomenti
Fondamentale per debugging. Il log deve contenere: nome tool, input, output, durata, eventuali errori. Senza questo, è impossibile capire perché l'agente ha fatto una scelta sbagliata.
MUST
Hai definito alert per tool call fallite o con latenza anomala
Un server MCP lento o che restituisce errori degrada silenziosamente la qualità dell'agente. Monitora la latenza e il tasso di errore per ogni server MCP in produzione.
SHOULD
