Le Variabili Discriminanti

// MOD-00

Prima di costruire le tassonomie, occorre identificare le variabili che creano distinzioni reali e rilevanti per chi deve scegliere un modello. Non tutte le variabili pesano allo stesso modo in tutti i contesti d'uso.

Le 8 variabili che contano davvero

V1
Capacità di ragionamento
Qualità del ragionamento multi-step, performance su task complessi, gestione dell'ambiguità. È la variabile più direttamente correlata alla qualità dell'output su task difficili. Proxy empirico: benchmark come MMLU, GPQA, SWE-bench, LMSYS Arena.
V2
Velocità (latenza)
Token al secondo, time-to-first-token. Critica per applicazioni interattive e agenti con molte chiamate in sequenza. I modelli grandi sono più lenti; i modelli piccoli e distillati sono spesso sorprendentemente veloci su task semplici.
V3
Costo per token
Input/output token price. Varia di 50–100x tra modelli frontier e modelli fast. Su volumi alti è la variabile economica dominante. Va sempre valutata in combinazione con V1: il costo reale è costo/(qualità task specifico).
V4
Context window
Lunghezza massima del contesto (input + output). Determina quanti documenti, quanta history, quante istruzioni si possono fornire in una singola chiamata. Oggi varia da 8K a 1M+ token. Critica per RAG, agenti con long context, analisi di documenti lunghi.
V5
Modalità di accesso
API proprietaria vs pesi scaricabili (open weights) vs self-hosted. Determina data privacy, costi su scala, possibilità di fine-tuning, dipendenza dal vendor. Per molte organizzazioni questa è la variabile non negoziabile prima di qualsiasi altra.
V6
Modalità input/output
Testo, immagini, audio, video, codice, tool use, function calling, output strutturato. I modelli solo-testo sono più semplici da integrare; i modelli multimodali aprono use case impossibili altrimenti. Il function calling affidabile è critico per gli agenti.
V7
Specializzazione di dominio
General purpose vs verticale (codice, medicina, diritto, scienze). I modelli general purpose sono più flessibili; i verticali superano i general su task specifici nel loro dominio ma performano peggio fuori. La specializzazione può avvenire via pre-training o fine-tuning post.
V8
Tipo di ragionamento
Standard (risposta diretta) vs extended thinking / chain-of-thought esplicito (il modello "pensa" prima di rispondere). I modelli "thinking" (o3, Claude 3.7 Sonnet Extended Thinking) sono significativamente più lenti e costosi ma superiori su ragionamento complesso.

// Principio guida

Non esiste un modello ottimale per tutti i task. La tassonomia serve a identificare quale variabile è critica per il tuo caso d'uso specifico, e quindi quale classe di modelli è il punto di partenza giusto.

Tassonomia per Capacità e Tier

// T1 // VAR: V1 + V2 + V3

La classificazione più pratica per chi deve scegliere un modello per un task. Si basa sul bilanciamento tra capacità di ragionamento, velocità e costo — le tre variabili che determinano il ROI per la maggior parte dei casi d'uso.

I Quattro Tier

TIER 1 — Frontier

FRONTIER

I modelli più capaci disponibili. Massima qualità su ragionamento complesso, coding avanzato, analisi sfumata, task che richiedono comprensione profonda. Usati dove la qualità è non negoziabile e il costo è secondario o il volume è basso.

Esempi attuali: Claude Opus 4, GPT-4o, Gemini 1.5 Ultra, Llama 3.1 405B.

Capacità ragionamento
95/100
Velocità (token/s)
30/100
Convenienza costo
15/100
Context window
200k+
// Quando usarli

Task complessi che richiedono ragionamento profondo, analisi multi-step, orchestrazione di agenti, task critici dove un errore ha costo elevato. Non usarli per task routinari ad alto volume.

TIER 2 — Balanced

BALANCED

Il "sweet spot" per la maggior parte dei task in produzione. Ottima qualità su task standard, velocità accettabile, costo gestibile. È il tier da cui iniziare quasi sempre quando si costruisce un agente.

Esempi attuali: Claude Sonnet 4.5, GPT-4o mini (nelle versioni più capaci), Gemini 1.5 Pro, Llama 3.1 70B, Mistral Large.

Capacità ragionamento
78/100
Velocità (token/s)
65/100
Convenienza costo
60/100
Context window
128k–200k

TIER 3 — Fast / Light

FAST

Modelli ottimizzati per velocità e costo. Performance ridotta rispetto ai tier superiori, ma sufficiente per task ben definiti, classificazione, estrazione semplice, interazioni conversazionali standard. Ideali come worker in sistemi multi-agent dove l'orchestratore è Tier 1/2.

Esempi attuali: Claude Haiku 3.5, GPT-4o mini, Gemini 2.0 Flash, Llama 3.1 8B, Mistral 7B, Phi-4.

Capacità ragionamento
52/100
Velocità (token/s)
90/100
Convenienza costo
88/100
Context window
8k–128k

TIER 4 — Reasoning / Thinking

THINKING

Una categoria emersa nel 2024–2025. Modelli che producono un "pensiero interno" esteso prima di rispondere. Superiori su ragionamento matematico, logica formale, coding complesso, problem solving multi-step. Molto più lenti e costosi dei modelli standard, anche frontier.

Esempi attuali: OpenAI o1 / o3, Claude 3.7 Sonnet (Extended Thinking), DeepSeek R1, Gemini 2.0 Flash Thinking, QwQ-32B.

// Attenzione

Non usare i modelli thinking per task semplici — il costo e la latenza sono sproporzionati. Sono giustificati solo quando la qualità del ragionamento è la variabile critica e non c'è vincolo di tempo o budget severo.

Tassonomia per Modalità di Accesso

// T2 // VAR: V5

La variabile di accesso è spesso la più vincolante in contesti enterprise. Determina privacy dei dati, costi su scala, possibilità di customizzazione, e dipendenza dal vendor. Va valutata prima delle altre.

MODELLI LLM // CLASSIFICAZIONE PER ACCESSO
API PROPRIETARIE (closed source)
Pesi non accessibili — solo interfaccia API dati al vendor
Pay-per-token — nessun costo fisso infrastruttura setup rapido
Fine-tuning limitato o assente (Anthropic: no; OpenAI: sì su modelli selezionati)
SLA, supporto enterprise, compliance certificazioni (SOC2, HIPAA ecc.)
Esempi: Claude (Anthropic), GPT-4o (OpenAI), Gemini (Google), Command R (Cohere)
OPEN WEIGHTS (pesi scaricabili)
Pesi scaricabili — eseguibile on-premise o su infrastruttura propria privacy totale
Fine-tuning possibile su dati propri massima customizzazione
Costo fisso infrastruttura (GPU) — conveniente ad alto volume complessità ops
Licenze variabili: alcune commerciali (Llama 3), alcune Apache 2.0 (Mistral 7B, Falcon)
Esempi: Llama 3.1 (Meta), Mistral/Mixtral, Qwen 2.5 (Alibaba), Phi-4 (Microsoft), Gemma (Google)
API DI TERZE PARTI (hosting open weights)
Modelli open hostati da provider (Together AI, Fireworks, Groq, AWS Bedrock) dati a provider terzo
Più veloce del self-hosting, meno privacy del self-hosting
Utile per prototipare con open weights senza infrastruttura propria
Esempi: Llama via Groq, Mistral via Together AI, Llama via AWS Bedrock
FULLY OPEN SOURCE (pesi + codice + dati training)
Rari — pochi modelli rilasciano anche i dati di training massima trasparenza
Importanti per ricerca, audit, compliance in settori regolati
Esempi: OLMo (Allen AI), alcuni modelli Falcon (TII), BLOOM

Matrice decisionale accesso

Criterio API Proprietaria Open Weights Self-hosted API Terze Parti
PRIVACY DATI Dati al vendor — policy dipende da contratto Dati restano interni — privacy totale Dati al provider hosting
SETUP TIME Minuti — API key e call Giorni/settimane — infrastruttura GPU Ore — account + API key
COSTO BASSO VOLUME Ottimo — pay per use Costoso — GPU anche senza uso Buono — pay per use
COSTO ALTO VOLUME Costoso — scala linearmente Ottimo — costo marginale vicino a zero Medio

COMPLIANCE / AUDIT Certificazioni enterprise disponibili Controllo totale — ideale per settori regolati Varia per provider
DIPENDENZA VENDOR Alta — pricing e policy possono cambiare Nulla — portabilità totale Media

Tassonomia per Modalità Input/Output

// T3 // VAR: V6

La modalità di input e output determina quali task sono possibili. I modelli language-only restano la maggioranza, ma la multimodalità è in rapida espansione e apre use case che prima erano inaccessibili agli LLM.

Text-Only

BASE

Input e output solo testuali. Il paradigma originale. Ancora dominante per la maggior parte dei task enterprise. Più semplice da integrare, più prevedibile nel comportamento.

Esempi: vecchie versioni GPT-3/4, Mistral, molti modelli open

Vision + Text (Multimodale)

MULTIMODALE

Input: testo + immagini. Output: testo. Apre use case su documenti scansionati, analisi di grafici, comprensione di UI, descrizione di immagini, estrazione da PDF non parsabili.

Esempi: Claude 3+, GPT-4o, Gemini 1.5, Llama 3.2 Vision

Audio + Text

AUDIO

Input: audio/voce. Output: testo (e a volte audio). Critico per interfacce vocali, trascrizione e analisi di call, assistenti voice-first. Spesso richiede pipeline separata (Whisper + LLM).

Esempi: GPT-4o (nativo), Gemini 2.0 (nativo), altrimenti Whisper + any LLM

Tool Use / Function Calling

AGENTICI

Il modello può chiamare funzioni esterne in modo strutturato. Fondamentale per gli agenti. La qualità del function calling varia significativamente tra modelli — è una variabile critica da testare su task agentici.

Eccellenti: Claude 3+, GPT-4o. Buoni: Llama 3.1+, Mistral Large. Variabile: modelli più piccoli

Omnimodal (in evoluzione)

FRONTIER 2025+

Input: testo, immagini, audio, video, documenti. Output: testo, immagini (alcuni), audio (alcuni). Il paradigma verso cui si stanno muovendo i modelli frontier. GPT-4o e Gemini 2.0 sono i casi più avanzati. La generazione nativa di immagini e audio da LLM è ancora emergente.

// Nota pratica

Per la maggior parte degli use case agentici enterprise oggi, text + vision + function calling è la combinazione rilevante. Audio e video nativi restano casi d'uso di nicchia o in fase sperimentale.

Tassonomia per Architettura Interna

// T4 // VAR: V7 + V8

La tassonomia più tecnica — meno rilevante per chi sviluppa agenti (è quasi sempre una black box), ma importante per capire perché certi modelli si comportano in certi modi e dove hanno i loro limiti strutturali.

Per tipo di ragionamento

Standard (auto-regressive)

Il paradigma dominante. Il modello genera un token alla volta, senza revisione esplicita. Il "ragionamento" emerge come pattern nel testo generato. Veloce, efficiente, ma limitato su task che richiedono pianificazione profonda o backtracking.

Esempi: praticamente tutti i modelli pre-2024

Extended Thinking / Chain-of-Thought

Il modello produce un "scratchpad" interno di ragionamento prima della risposta finale. Il token budget del thinking è separato. Superiore su ragionamento formale, matematica, coding complesso. Costo e latenza molto più alti.

Esempi: o1/o3 (OpenAI), Claude 3.7 Extended Thinking, DeepSeek R1, QwQ

Per struttura del modello

Architettura Descrizione Vantaggio Svantaggio Esempi
Dense Transformer Tutti i parametri attivi ad ogni forward pass. Architettura standard. Semplice, prevedibile, ben compreso Costo computazionale scala con #parametri Llama, Mistral 7B, GPT-3
Mixture of Experts (MoE) Solo un sottoinsieme di "esperti" si attiva per ogni token. Parametri totali alti, parametri attivi bassi. Efficienza computazionale alta, buona qualità per i FLOP spesi Più complesso da deployare, spesso incoerente su task che richiedono tutti gli esperti Mixtral 8x7B, Mixtral 8x22B, GPT-4 (presunto), Gemini 1.5
State Space Models (SSM) Alternativa al transformer per sequenze lunghe. Attenzione lineare invece di quadratica. Efficiente su context window lunghissime, memoria costante Meno capability sui task standard rispetto ai transformer equivalenti Mamba, Jamba (AI21)
Hybrid (SSM + Transformer) Combina layer SSM e layer di attenzione tradizionale. Efficienza su sequenze lunghe + capability transformer Architettura emergente, meno matura Jamba 1.5, alcuni modelli ricerca

Per specializzazione

General Purpose

Training su dati eterogenei — testo, codice, scienze, cultura. Massima flessibilità, nessuna specializzazione. Il 95% dei modelli rilevanti rientra in questa categoria.

Domain-Specific

Pre-training o fine-tuning su dati di un dominio specifico. Supera i general purpose nel dominio, performance peggiore fuori. Esempi: CodeLlama (coding), BioMedLM (biomedica), LexLM (legal).

Task-Specific / Instruct

Fine-tuning per seguire istruzioni o un task specifico (classificazione, sintesi, traduzione). Spesso modelli piccoli + fine-tuning battono modelli grandi general purpose su task narrowi e ben definiti.

Tassonomia per Uso Pratico

// T5 // VAR: COMPOSITA

La tassonomia più utile per chi sviluppa agenti. Classifica i modelli non per caratteristica tecnica ma per il ruolo che ricoprono in un sistema agentivo. Un buon sistema usa spesso modelli diversi per ruoli diversi.

Ruoli in un sistema agentivo

R1
Orchestratore / Planner
Riceve l'obiettivo, scompone in sotto-task, decide quale agente/tool attivare, gestisce gli errori. Deve essere il modello più capace disponibile — gli errori di pianificazione si amplificano su tutta la pipeline. Tier 1 o Thinking — non fare economia qui.
R2
Worker Generalista
Esegue task di media complessità: generazione testo, analisi documenti, Q&A su knowledge base, coding standard. Il "cavallo di battaglia" della maggior parte delle pipeline. Tier 2 (Balanced) — il miglior rapporto qualità/costo nella maggior parte dei casi.
R3
Worker Veloce / High-Volume
Gestisce task semplici e ripetitivi ad alto volume: classificazione, routing, estrazione semplice, triaging. La velocità e il costo basso sono le variabili critiche, non la qualità assoluta. Tier 3 (Fast) — in alcuni casi un modello fine-tunato piccolo supera un modello grande general purpose sul task specifico.
R4
Validator / Critic
Valuta l'output di altri agenti rispetto a criteri espliciti. Deve essere indipendente dal modello che ha generato l'output da valutare (per evitare bias di auto-conferma). Spesso lo stesso tier del worker che valuta, o superiore su task critici. Tier 2+
R5
Reasoner / Solver
Task che richiedono ragionamento profondo: matematica, logica formale, debugging complesso, analisi di casi edge. Giustifica i modelli thinking anche con i loro costi. Tier 4 (Thinking) — ma solo quando davvero necessario.

Framework di selezione in 5 domande

Domanda Se SÌ Se NO
I dati possono uscire dall'org? API proprietaria è accettabile Self-hosted open weights obbligatorio
Il task richiede ragionamento complesso? Tier 1 o Thinking Tier 2 o Tier 3 — non sprecare budget
Il volume è alto (>1M token/giorno)? Tier 3 o self-hosted — il costo conta API proprietaria, Tier 2 va bene
Il task richiede immagini o audio? Modello multimodale (Claude 3+, GPT-4o) Modello text-only, spesso più economico
Serve function calling affidabile? Claude 3+, GPT-4o — testati e robusti Qualsiasi tier è accettabile
